SWU 이메일 수집 자동화 최종 설계서

문서명      : SWU 이메일 수집 자동화 최종 설계서
작성일      : 2026-05-02
작성자      : 고승현
버전        : v1.0
문서 구분   : 개발용 설계서
최종 산출물 : 전체 수집 결과 엑셀 1개
비고        : 고객 납품용 정리본은 이번 개발 범위에서 제외하고, 추후 별도 작업한다.


1. 기본 기준

    1-1. 수집 목적

        1-1-1. 목적

            1) 서울여자대학교 공개 페이지에서 이름과 이메일 정보를 수집한다.

            2) 수집 대상은 아래와 같다.

                대학
                대학원
                연구소
                부속기관
                산학협력단

            3) 최종 데이터는 object list 형태로 수집한 뒤 엑셀로 저장한다.


    1-2. 최종 결과 형태

        1-2-1. 데이터 구조

            1) 프로그램 내부에서는 데이터를 object list 형태로 관리한다.

            2) 각 object는 한 명 또는 한 개 이메일 정보를 의미한다.

        1-2-2. 저장 파일

            1) 최종 엑셀 파일은 2개

            2) 1은 전체 엑셀

            3) 2는 빈값과 중복이 제거된 엑셀 중복 기준은 첫 데이터만 남고 나머지 이메일이 동일한거는 모두 제거한다.


    1-3. 최종 저장 필드

        1-3-1. 엑셀 컬럼

            대분류
            중분류
            소분류
            직위
            업무
            이름
            이메일
            URL
            홈페이지URL

        1-3-2. 저장 object 예시

            {
                "대분류": "인문대학",
                "중분류": "국어국문학과",
                "소분류": "국어국문학과",
                "직위": "교수",
                "업무": "교육·연구",
                "이름": "홍길동",
                "이메일": "sample@swu.ac.kr",
                "URL": "https://www.swu.ac.kr/www/liberala_1.html",
                "홈페이지URL": "https://korean.swu.ac.kr"
            }


    1-4. 제외 필드

        1-4-1. 전화번호

            1) 전화번호는 수집하지 않는다.

            2) 테이블에 전화번호 컬럼이 있어도 저장하지 않는다.


    1-5. 업무 저장 기준

        1-5-1. 교수

            직위 = 교수
            업무 = 교육·연구

        1-5-2. 행정실

            직위 = 빈값
            업무 = 행정
            이름 = 빈값

        1-5-3. 직원 / 팀장 / 연구원 / 조교

            1) 페이지에 직위가 있으면 해당 직위를 그대로 저장한다.

            2) 페이지에 업무가 있으면 해당 업무를 그대로 저장한다.


    1-6. 코드 작성 규칙

        1-6-1. 기본 방향

            1) 코드는 쉽고 짧고 명확하게 작성한다.

            2) 불필요하게 복잡한 구조는 만들지 않는다.

            3) 불필요한 함수 분리는 하지 않는다.

            4) 불필요한 검증 코드는 넣지 않는다.

        1-6-2. 제외할 코드 스타일

            예외처리 최소화
            타입 체크 사용 안 함
            최신 문법 사용 안 함
            어려운 문법 사용 안 함

        1-6-3. 로그 기준

            1) 현재 어느 단계가 실행 중인지 알 수 있게 로그를 출력한다.

            2) 현재 몇 번째 데이터를 처리 중인지 알 수 있게 로그를 출력한다.

            3) 로그는 너무 많지 않게, 진행 상황 확인이 가능한 수준으로만 넣는다.

    1-7. 개발 방식

        1-7-1. 사용 기술

            Python
            Selenium
            BeautifulSoup
            pandas 또는 openpyxl

        1-7-2. 브라우저 실행 기준

            1) headless 모드로 실행한다.

            2) 기본적으로 time.sleep은 넣지 않는다.

            3) 필요한 경우에만 최소한으로 time.sleep을 추가한다.

        1-7-3. 속도 기준

            1) 공개 페이지 기준으로 최대한 빠르게 수집한다.

            2) 불필요한 페이지 이동은 줄인다.

            3) 이미 수집한 URL은 중복 접속하지 않도록 한다.

            4) 셀레니움은 느린 경우가 많으니 속도 최적화를 꼭 한다.


    1-8. 엑셀 저장 기준

        1-8-1. 파일명

            1) 엑셀 파일명에는 실행 시점의 yyyymmddhhmmss 값을 붙인다.

            2) 파일명 예시는 아래와 같다.

                SWU_EMAIL_RESULT_20260502183000.xlsx

        1-8-2. 저장 개수

            1) 이번 프로그램에서는 최종본 엑셀 1개만 생성한다.

            2) 이메일 빈값 제거, 중복 이메일 제거 같은 고객용 정리 작업은 이번 범위에서 제외한다.


    1-9. 홈페이지 접속 기준

        1-9-1. 외부 도메인 접속

            1) 학과 홈페이지처럼 외부 도메인으로 이동하는 URL은 접속이 안 될 수 있다.

            2) DNS 오류 또는 접속 실패가 생길 수 있다.

        1-9-2. 예외처리 기준

            1) 홈페이지 URL 접속 실패 부분만 최소 예외처리한다.

            2) 접속 실패 시 프로그램 전체가 종료되지 않게 한다.

            3) 접속 실패한 홈페이지URL은 로그로 남긴다.


2. 대학 데이터 수집

    2-1. 기준 URL

        2-1-1. 대학 목록 기준 URL

            https://www.swu.ac.kr/www/swuniversity.html


    2-2. 대학 / 학과 목록 수집

        2-2-1. 기본 수집 영역

            1) 기준 URL에 접속한다.

            2) #main 안의 .section 목록을 찾는다.

            3) .section 목록을 반복한다.

        2-2-2. 대분류 추출 기준

            1) 각 .section 안에서 .titl1 또는 .titl0 영역을 찾는다.

            2) .titl1 또는 .titl0 텍스트를 대분류로 사용한다.

        2-2-3. 경우 1 - 제목 영역에 a 링크가 바로 있는 경우

            1) .titl1 또는 .titl0 안에 a 링크가 바로 있으면 해당 링크를 수집한다.

            2) a 태그의 text를 대분류, 중분류, 소분류에 모두 넣는다.

            3) a 태그의 href를 URL로 저장한다.

            예시 HTML

                <div class="titl1">
                    <a href="/www/fundaedu_1.html">교양대학</a>
                </div>

            저장 결과

                대분류 = 교양대학
                중분류 = 교양대학
                소분류 = 교양대학
                URL    = https://www.swu.ac.kr/www/fundaedu_1.html

        2-2-4. 경우 2 - 제목 아래에 ul.col_list0 이 있는 경우

            1) .titl1 또는 .titl0 텍스트를 대분류로 사용한다.

            2) ul.col_list0 li a 목록을 찾는다.

            3) li a 텍스트를 중분류와 소분류로 사용한다.

            4) li a의 href를 URL로 저장한다.

            저장 결과 예시

                대분류 = 인문대학
                중분류 = 국어국문학과
                소분류 = 국어국문학과
                URL    = https://www.swu.ac.kr/www/liberala_1.html


    2-3. 전공 탭 처리

        2-3-1. 학과 페이지 접속

            1) 2-2에서 수집한 학과 URL에 접속한다.

            2) 학과 페이지 안에서 class="tabui4" 영역을 찾는다.

        2-3-2. 전공 탭이 있는 경우

            1) class="tabui4" 가 있으면 해당 학과가 여러 전공으로 나뉘어 있는 것으로 본다.

            2) tabui4 안의 a 링크를 모두 수집한다.

            3) 각 a 링크의 text는 전공명으로 사용한다.

            4) 각 a 링크의 href는 전공 URL로 저장한다.

            5) 수집한 전공 URL에 하나씩 접속한다.

            6) 각 전공 페이지에서 다시 "교수진"문구를 포함한 링크를 찾는다.

            저장 기준

                대분류 = 기존 대학명
                중분류 = 전공명
                소분류 = 전공명
                URL    = 전공 URL

        2-3-3. 전공 탭이 없는 경우

            1) class="tabui4" 가 없으면 현재 학과 페이지에서 바로 "교수진"문구를 포함한 링크를 찾는다.

            저장 기준

                대분류 = 기존 대학명
                중분류 = 기존 학과명
                소분류 = 기존 학과명
                URL    = 기존 학과 URL


    2-4. 홈페이지 URL 수집

        2-4-1. 수집 위치

            1) 학과 또는 전공 페이지 안에서 "홈페이지"를 포함한 버튼을 찾는다.

        2-4-2. 수집 태그 기준

            1) class="btn btn_xl btn_blue_gray" 인 a 태그를 찾는다.

            2) 단, 해당 a 태그의 텍스트에 "홈페이지" 문구가 포함된 경우만 수집한다.

            3) 해당 a 태그의 href 값을 홈페이지URL로 저장한다.

        2-4-3. 텍스트 판단 기준

            1) 버튼 텍스트는 페이지마다 다를 수 있다.

            2) 정확히 일치하는 방식으로 비교하지 않는다.

            3) "홈페이지" 포함 여부로 판단한다.

            텍스트 예시

                홈페이지
                홈페이지 바로가기
                홈페이지바로가기
                홈페이지 이동

        2-4-4. 홈페이지URL이 여러 개인 경우

            1) 홈페이지URL이 여러 개 있으면 모두 확인 대상으로 포함한다.

            2) 각 홈페이지URL에 접속해서 이메일을 찾는다.


    2-5. 교수진 페이지 찾기

        2-5-1. 교수진 링크 영역

            1) 학과 또는 전공 페이지 안에서 class="tabui0" 영역을 찾는다.

        2-5-2. 교수진 링크 판단

            1) tabui0 안의 a 태그 중 text가 "교수진" 인 링크를 찾는다.

            2) 찾으면 해당 링크로 이동한다.


    2-6. 교수 정보 수집

        2-6-1. 교수 목록 영역

            1) 교수진 페이지에서 ul.pro_list를 찾는다.

            2) ul.pro_list 안의 li를 반복한다.

        2-6-2. 이름 수집

            1) 이름은 .CELL.CELL1 안의 .name에서 가져온다.

            2) 이름에 직위 문구가 있으면 제거한다.

            제거 대상 문구

                교수
                조교수
                부교수
                명예교수
                석좌교수
                특임교수

            3) 이름 안의 불필요한 공백도 제거한다.

        2-6-3. 이메일 수집

            1) 이메일은 .info 안의 mailto 링크에서 가져온다.

            2) mailto: 부분은 제거하고 이메일 주소만 저장한다.


    2-7. 교수 데이터 저장 규칙

        2-7-1. 저장 값

            대분류      = 대학명
            중분류      = 학과명 또는 전공명
            소분류      = 학과명 또는 전공명
            직위        = 교수
            업무        = 교육·연구
            이름        = 교수 이름
            이메일      = mailto 이메일
            URL         = 교수진 페이지 URL
            홈페이지URL = 홈페이지 바로가기 URL


    2-8. 행정실 이메일 수집

        2-8-1. 수집 시점

            1) 교수 정보 수집이 끝난 뒤 홈페이지URL에 접속한다.

        2-8-2. 이메일 검색 범위

            1) 홈페이지 첫 화면의 전체 HTML에서 이메일 형식을 찾는다.

            2) footer 안에 있을 수도 있고 다른 위치에 있을 수도 있으므로 페이지 전체에서 찾는다.

        2-8-3. 이메일 정규식 기준

            1) @swu.ac.kr 만 찾지 않는다.

            2) 일반 이메일 형식이면 모두 수집한다.

            이메일 예시

                sample@swu.ac.kr
                sample@gmail.com
                sample@naver.com

        2-8-4. 이메일 예외 기준

            wixpress.com, sentry.io 2개중 1개라도 포함하면 해당 이메일은 추가하지 않는다.


    2-9. 행정실 데이터 저장 규칙

        2-9-1. 저장 값

            대분류      = 대학명
            중분류      = 학과명 또는 전공명
            소분류      = 행정실
            직위        = 빈값
            업무        = 행정
            이름        = 빈값
            이메일      = 홈페이지에서 찾은 이메일
            URL         = 학과 또는 전공 URL
            홈페이지URL = 홈페이지 바로가기 URL


    2-10. 행정실 예외 추가 case - 교직지원센터

        2-10-1. 기준 URL

            https://sweet.swu.ac.kr/skin/page/about03.html

        2-10-2. 수집 시작 영역

            1) 페이지에 접속한다.

            2) class="se2 col-md-12" 영역을 시작점으로 잡는다.

            3) 해당 영역은 교직지원센터 직원 정보 영역이다.

            4) 이 페이지에서 유일한 기준 영역으로 사용한다.

        2-10-3. 수집 상세 영역

            1) class="se2 col-md-12" 영역 안에서 class="row profile-wrap" 영역을 찾는다.

            2) profile-wrap 안에서 class="caption" 목록을 찾는다.

            3) 각 caption 안에서 아래 정보를 수집한다.

                p.name  = 이름 + 직위
                p.area  = 업무
                p.email = 이메일

        2-10-4. 이름 / 직위 분리 기준

            1) p.name 값은 이름과 직위가 함께 들어 있다.

            2) 마지막 공백 뒤의 단어를 직위로 본다.

            3) 앞쪽 텍스트를 이름으로 본다.

            예시

                홍광오 팀장
                조효상 차장
                유지원 원생조교

            저장 예시

                원본 = 홍광오 팀장
                이름 = 홍광오
                직위 = 팀장

        2-10-5. 이메일 수집 기준

            1) p.email 텍스트에서 이메일 형식만 추출한다.

            예시

                E-mail : hko@swu.ac.kr

            저장 결과

                이메일 = hko@swu.ac.kr

        2-10-6. 저장 규칙

            대분류      = 교양대학
            중분류      = 교양대학
            소분류      = 교직지원센터
            직위        = p.name에서 분리한 직위
            업무        = p.area 텍스트
            이름        = p.name에서 분리한 이름
            이메일      = p.email에서 추출한 이메일
            URL         = https://sweet.swu.ac.kr/skin/page/about03.html
            홈페이지URL = https://sweet.swu.ac.kr/skin/page/about03.html

        2-10-7. 처리 위치

            1) 일반 대학 교수 정보 수집을 먼저 실행한다.

            2) 일반 대학 행정실 이메일 수집을 실행한다.

            3) 그 다음 교직지원센터 예외 수집을 별도로 한 번 실행한다.

        2-10-8. 주의사항

            1) 전화번호는 수집하지 않는다.

            2) 반드시 class="se2 col-md-12" 영역을 시작점으로 삼는다.

            3) 이 영역 밖의 caption 정보는 수집하지 않는다.

            4) 이 예외 케이스는 일반 행정실 이메일 수집 방식과 다르다.

            5) 이름, 직위, 업무, 이메일을 모두 수집한다.


    2-11. 대학 예외 추가 case

6. 예외 처리 기준

    6-1. 자동 수집 예외

        6-1-1. 제외 기준

            1) 대학, 대학원, 연구소, 부속기관, 산학협력단 흐름으로 해결되지 않는 데이터는 자동 수집 대상에서 제외한다.

            2) 구조가 명확하지 않은 페이지는 억지로 수집하지 않는다.


    6-2. 수작업 확인 대상

        6-2-1. 수작업 대상

            1) 자동 수집으로 해결되지 않는 경우 개발자가 직접 홈페이지에 들어가서 확인한다.

            2) 홈페이지 접속 실패가 발생한 경우 수작업 확인 대상으로 남긴다.

            3) 페이지 구조가 설계 기준과 다른 경우 수작업 확인 대상으로 남긴다.


    6-3. 홈페이지 접속 예외

        6-3-1. 발생 가능 상황

            홈페이지 URL 열리지 않음
            DNS 오류
            외부 도메인 연결 실패
            페이지 로딩 실패

        6-3-2. 처리 방식

            1) 해당 홈페이지는 건너뛴다.

            2) 프로그램 전체는 종료하지 않는다.

            3) 실패한 홈페이지URL을 로그로 남긴다.


    6-4. 페이지 구조 예외

        6-4-1. 예외 기준

            1) 페이지 구조가 설계서 기준 태그와 다르면 자동 수집하지 않는다.

            2) 개발자가 직접 페이지를 확인한다.

            3) 필요한 경우 별도 예외 로직을 추가한다.


    6-5. 이메일 예외

        6-5-1. 이메일이 없는 경우

            1) 이메일이 없으면 이메일 컬럼은 빈값으로 저장한다.

            2) 이번 최종본에서는 이메일 빈값도 그대로 남긴다.

        6-5-2. 이메일이 중복되는 경우

            1) 중복 이메일이 있어도 이번 최종본에서는 제거하지 않는다.

            2) 중복 제거는 고객 납품용 정리본 작업 때 별도로 처리한다.

    7-1. 실행 순서

        7-1-1. 1단계 - 대학 데이터 수집

        7-1-2. 2단계 - 대학원 데이터 수집

            1) 대학원 교학팀 데이터를 수집한다.

            3) 대학원 학과 또는 전공 목록을 수집한다.

            5) 대학원 교수 정보를 수집한다.

        7-1-3. 3단계 - 연구소 / 부속기관 데이터 수집

            1) 부설연구기관 목록을 수집한다.

            2) 각 기관의 구성원 페이지를 찾는다.

            3) 교수 목록이 있으면 교수 정보를 수집한다.

            4) 테이블 정보가 있으면 직원, 연구원, 조교 정보를 수집한다.

            1) 산학협력단 페이지에 접속한다.

            2) table-wrap 목록을 반복한다.

            3) 조직명과 인원 정보를 수집한다.

        7-1-5. 5단계 - 최종 엑셀 저장

            1) 모든 수집 데이터를 하나의 object list로 합친다.

            2) 최종본 엑셀 1개로 저장한다.

            3) 이메일 빈값과 중복 이메일은 제거하지 않는다.
