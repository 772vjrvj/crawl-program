개정 내용

    1) 기존 간단 설명서를 현재 통합 py 기준으로 상세화하였다.

    2) 대학.py, 대학원.py, 산학.py 3개 파트를 하나의 통합 py에서 실행하는 구조를 설명하였다.

    3) 원본 엑셀과 정리본 엑셀의 차이를 명확하게 정리하였다.

    4) 이메일 빈값 제거, 이메일 형식 검증, 이메일 중복 제거 기준을 상세하게 정리하였다.

    5) 대학 / 대학원 / 산학협력단별 크롤링 대상, URL, HTML selector, 예외 case를 세부적으로 정리하였다.

    6) 광고홍보학과, 의료인공지능공학과, 산업시스템공학과, 영상/영화 학과, 사회복지상담학과, 글로벌무역학과 예외 수집 기준을 포함하였다.

    7) 공지사항 외 학사공지, 학부공지, 학사정보, 학과소식, 알림판, 학과사무실공지, 게시판 메뉴 탐색 기준을 포함하였다.

    8) 나중에 AI에게 이 설명서만 전달해도 동일한 목적의 코드를 다시 만들 수 있도록 단계별 처리 기준을 보강하였다.


1. 전체 목적

    1-1. 목적

        1-1-1. 기본 목적

            1) 동국대학교 공개 페이지에서 교수 이메일과 행정 이메일을 수집한다.

            2) 수집 대상은 크게 아래 3개 영역이다.

                대학
                대학원
                산학협력단

            3) 기존에 분리되어 있던 아래 3개 py 파일을 하나의 통합 실행 파일로 연결한다.

                대학.py
                대학원.py
                산학.py

            4) 통합 실행 파일 하나만 실행하면 3개 영역을 순서대로 수집하고, 최종 엑셀 2개를 생성한다.


        1-1-2. 최종 목표

            1) 작업자 보관용 원본 엑셀을 만든다.

            2) 고객 전달 또는 후처리용 정리본 엑셀을 만든다.

            3) 원본 엑셀은 수집 결과를 최대한 그대로 보존한다.

            4) 정리본 엑셀은 이메일 기준으로 사용할 수 있는 데이터만 남긴다.


    1-2. 수집 범위

        1-2-1. 대학

            1) 동국대학교 대학 목록 페이지에서 단과대학 / 학과 목록을 수집한다.

            2) 각 학과 메인 URL에 접속한다.

            3) 교수진 / 교수소개 / 전임교원 등 교수 메뉴를 찾아 교수 이름과 이메일을 수집한다.

            4) 교수 수집 후 학과 메인에서 공지/게시판 메뉴를 찾는다.

            5) 공지/게시판 상세 페이지에서 행정 이메일을 찾는다.

            6) 공지에서 찾은 이메일이 교수 이메일과 겹치면 제외한다.

            7) 교수 이메일과 겹치지 않는 @dongguk 계열 이메일을 찾으면 소분류를 행정으로 저장한다.


        1-2-2. 대학원

            1) 동국대학교 대학원 목록을 수집한다.

            2) 대학원별 사이트에 접속한다.

            3) 학과안내 / 학과(전공) / 전공안내 메뉴에서 학과 목록을 수집한다.

            4) 학과가 있으면 학과 메인 페이지에서 행정 이메일을 수집한다.

            5) 학과가 없으면 대학원 메인 페이지에서 행정 이메일을 수집한다.

            6) 교수소개 / 교수진 소개 / 교수진 메뉴에서 교수 이름과 이메일을 수집한다.


        1-2-3. 산학협력단

            1) 산학협력단 기준 페이지에 접속한다.

            2) 조직도 메뉴는 제외한다.

            3) 부서별 페이지에서 담당업무와 이메일을 수집한다.

            4) 수집한 이메일은 행정 데이터로 저장한다.


    1-3. 제외 범위

        1-3-1. 제외 항목

            1) 로그인해야 볼 수 있는 비공개 데이터는 수집하지 않는다.

            2) 전화번호는 최종 저장 컬럼에 포함하지 않는다.

            3) 이미지 안에 들어 있는 이메일은 OCR로 읽지 않는다.

            4) 첨부파일 안의 이메일은 현재 수집하지 않는다.

            5) PDF, HWP, DOCX 첨부 파일 내부 이메일은 현재 수집하지 않는다.

            6) 고객용 컬럼명 재가공이나 별도 양식 변환은 현재 범위에서 제외한다.


2. 최종 산출물 기준

    2-1. 통합 실행 파일

        2-1-1. 파일명

            dongguk_email_all_in_one.py


        2-1-2. 역할

            1) 대학 / 대학원 / 산학협력단 수집 코드를 통합 실행한다.

            2) 내부적으로 3개 파트 코드를 임시 py 파일로 생성한다.

            3) 생성된 임시 py 파일을 순서대로 실행한다.

            4) 각 파트에서 생성된 중간 엑셀을 읽는다.

            5) 최종 원본 엑셀과 정리본 엑셀을 생성한다.


    2-2. 중간 산출물

        2-2-1. 중간 엑셀

            1) 대학 파트 결과

                동국대_교수목록.xlsx

            2) 대학원 파트 결과

                동국대_대학원_교수목록.xlsx

            3) 산학협력단 파트 결과

                동국대_산학협력단_행정.xlsx


        2-2-2. 중간 임시 폴더

            1) 통합 py는 실행 시 아래 임시 폴더를 만든다.

                동국대_통합_임시

            2) 임시 폴더 안에는 아래 3개 파트 파일이 생성된다.

                dongguk_part_college.py
                dongguk_part_graduate.py
                dongguk_part_sanhak.py

            3) 통합 py 안의 PARTS 배열에 들어 있는 source_b64 값을 복원하여 위 파일을 생성한다.


    2-3. 최종 원본 엑셀

        2-3-1. 파일명

            동국대_전체_원본_yyyymmddhhmmss.xlsx


        2-3-2. 용도

            1) 작업자 보관용 전체 수집 결과이다.

            2) 수집 과정에서 나온 데이터를 최대한 그대로 보존한다.

            3) 나중에 수작업 검수할 row도 남겨둔다.


        2-3-3. 저장 기준

            1) 이메일이 있는 row를 저장한다.

            2) 이메일이 없는 row도 저장한다.

            3) 이메일 형식이 아닌 값이 들어 있어도 원본에는 그대로 저장한다.

            4) 이메일이 중복되어도 원본에는 그대로 저장한다.

            5) 교수 목록을 못 찾은 경우 구분용 빈 row도 저장한다.

            6) 자동 수집 실패 또는 URL 없음 표시용 빈 row도 저장한다.

            7) 수집 순서를 최대한 유지한다.


    2-4. 최종 정리본 엑셀

        2-4-1. 파일명

            동국대_전체_정리본_yyyymmddhhmmss.xlsx


        2-4-2. 용도

            1) 이메일 기준으로 정리된 결과이다.

            2) 고객 전달 또는 후처리용으로 사용할 수 있다.

            3) 실제 이메일 형식만 남긴다.


        2-4-3. 저장 기준

            1) 이메일 빈값 row는 제거한다.

            2) 이메일 정규식에 맞지 않는 row는 제거한다.

            3) 이메일 앞뒤 공백과 불필요한 끝 문자를 제거한다.

            4) 이메일 비교는 소문자 기준으로 한다.

            5) 동일 이메일이 여러 번 나오면 가장 위에 있는 첫 번째 row만 남긴다.

            6) 두 번째 이후 같은 이메일 row는 모두 제거한다.


3. 최종 저장 컬럼

    3-1. 공통 컬럼

        3-1-1. 최종 컬럼 순서

            구분
            대분류
            중분류
            소분류
            직위
            업무
            이름
            이메일
            URL
            학과URL
            교수URL


    3-2. 컬럼 의미

        3-2-1. 구분

            1) 데이터 출처 영역을 저장한다.

            2) 값 예시는 아래와 같다.

                대학
                대학원
                산학협력단


        3-2-2. 대분류

            1) 대학 데이터에서는 단과대학명을 저장한다.

            2) 대학원 데이터에서는 대학원명을 저장한다.

            3) 산학협력단 데이터에서는 산학협력단을 저장한다.


        3-2-3. 중분류

            1) 대학 데이터에서는 학과명 또는 전공명을 저장한다.

            2) 대학원 데이터에서는 학과명 또는 대학원명을 저장한다.

            3) 산학협력단 데이터에서는 부서명을 저장한다.


        3-2-4. 소분류

            1) 교수 데이터에서는 보통 중분류와 동일하게 저장한다.

            2) 행정 이메일은 행정으로 저장한다.

            3) 산학협력단 데이터는 행정으로 저장한다.


        3-2-5. 직위

            1) 교수 데이터는 교수로 저장한다.

            2) 행정 이메일은 빈값으로 저장한다.

            3) 산학협력단은 현재 빈값으로 저장한다.


        3-2-6. 업무

            1) 교수 데이터는 교육·연구로 저장한다.

            2) 일반 행정 이메일은 행정으로 저장한다.

            3) 산학협력단은 표 안의 담당업무 텍스트를 저장한다.


        3-2-7. 이름

            1) 교수 데이터는 교수 이름을 저장한다.

            2) 행정 이메일은 빈값으로 저장한다.

            3) 산학협력단도 현재 이름은 저장하지 않는다.


        3-2-8. 이메일

            1) 수집한 이메일 주소를 저장한다.

            2) 원본에는 수집값을 최대한 그대로 둔다.

            3) 정리본에서는 clean_email 함수 기준으로 정규화된 이메일만 남긴다.


        3-2-9. URL

            1) 대학 교수 데이터는 학과 URL을 저장한다.

            2) 대학 공지 행정 데이터는 이메일을 찾은 공지 상세 URL을 저장한다.

            3) 대학원 데이터는 대학원 URL을 저장한다.

            4) 산학협력단 데이터는 부서 페이지 URL을 저장한다.


        3-2-10. 학과URL

            1) 대학원 학과가 있는 경우 학과 URL을 저장한다.

            2) 대학 파트에는 없는 컬럼이므로 통합 과정에서 빈값으로 채운다.

            3) 산학협력단도 빈값으로 채운다.


        3-2-11. 교수URL

            1) 교수 목록 페이지 URL을 저장한다.

            2) 행정 이메일은 빈값으로 저장한다.


4. 코드 작성 규칙

    4-1. 기본 원칙

        4-1-1. 코드 스타일

            1) 코드는 쉽고 짧고 명확하게 작성한다.

            2) 불필요하게 복잡한 구조는 만들지 않는다.

            3) 과한 클래스 구조는 만들지 않는다.

            4) 기존 코드 흐름과 변수명을 최대한 유지한다.

            5) 예외 case는 일반 수집 흐름을 망가뜨리지 않도록 별도 mapping 또는 전용 parser로 추가한다.


        4-1-2. 예외처리 기준

            1) 예외처리는 전체 프로그램이 끊기지 않는 수준으로만 최소 적용한다.

            2) Selenium 페이지 접속 실패, 알림창, 타임아웃 같은 실행 중단 가능 지점만 처리한다.

            3) 데이터가 없는 경우는 빈 row를 남겨 후속 검수 가능하게 한다.


        4-1-3. 로그 기준

            1) 현재 어느 단계가 실행 중인지 알 수 있게 로그를 출력한다.

            2) 현재 몇 번째 데이터를 처리 중인지 알 수 있게 로그를 출력한다.

            3) 로그는 너무 많지 않게 진행 상황 확인 가능한 수준으로만 출력한다.

            4) 주요 로그 예시는 아래와 같다.

                [시작] 동국대학교 이메일 통합 수집
                [실행] 대학 수집 시작
                [실행] 대학원 수집 시작
                [실행] 산학협력단 수집 시작
                [수집] 대분류 > 중분류 | 교수 N명
                [공지메일 발견] email url
                [엑셀] 원본 저장
                [엑셀] 정리본 저장
                [종료] 전체 작업 완료


    4-2. 사용 기술

        4-2-1. Python 패키지

            selenium
            pandas
            beautifulsoup4
            openpyxl
            requests


        4-2-2. Selenium 사용 목적

            1) 동국대 대학 / 대학원 사이트는 메뉴와 상세 이동에 Selenium이 필요한 경우가 있다.

            2) 교수 페이지와 공지 상세 페이지 이동에 Selenium을 사용한다.

            3) iBuilder 게시판의 javascript:fn_view 실행도 Selenium execute_script로 처리한다.


        4-2-3. requests 사용 목적

            1) 산학협력단은 정적 테이블 구조이므로 requests + BeautifulSoup으로 처리한다.

            2) Selenium보다 빠르게 처리하기 위해 산학협력단은 requests를 사용한다.


5. 통합 py 실행 구조

    5-1. 핵심 상수

        5-1-1. 이메일 정규식

            EMAIL_RE = [A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}


        5-1-2. 임시 폴더

            PART_DIR = 동국대_통합_임시


        5-1-3. 파트 목록

            PARTS 배열에 아래 3개 파트가 들어 있다.

                name = 대학
                filename = dongguk_part_college.py
                output = 동국대_교수목록.xlsx
                source_b64 = 대학.py 내용 base64

                name = 대학원
                filename = dongguk_part_graduate.py
                output = 동국대_대학원_교수목록.xlsx
                source_b64 = 대학원.py 내용 base64

                name = 산학협력단
                filename = dongguk_part_sanhak.py
                output = 동국대_산학협력단_행정.xlsx
                source_b64 = 산학.py 내용 base64


    5-2. main 실행 순서

        5-2-1. 전체 흐름

            1) [시작] 로그를 출력한다.

            2) delete_old_part_outputs 함수를 실행한다.

            3) write_part_files 함수를 실행한다.

            4) PARTS 배열을 반복하면서 run_part를 실행한다.

            5) read_part_excels 함수로 중간 엑셀 3개를 읽는다.

            6) save_final_excels 함수로 최종 원본 / 정리본 엑셀을 저장한다.

            7) [종료] 로그를 출력한다.


        5-2-2. delete_old_part_outputs

            1) 이전 실행에서 만들어진 중간 결과 엑셀이 있으면 삭제한다.

            2) 삭제 대상은 PARTS 배열의 output 값이다.

            3) 이전 중간 파일 때문에 새 실행 결과가 섞이는 것을 방지한다.


        5-2-3. write_part_files

            1) 동국대_통합_임시 폴더가 없으면 생성한다.

            2) 각 PART의 source_b64 값을 base64 decode 한다.

            3) 복원한 소스코드를 동국대_통합_임시 폴더에 py 파일로 저장한다.

            4) part["path"] 값에 생성된 py 경로를 넣는다.


        5-2-4. run_part

            1) runpy.run_path로 각 파트 py를 실행한다.

            2) run_name은 __main__으로 실행한다.

            3) 각 파트는 실행 후 자신의 중간 엑셀을 만든다.

            4) 파트 실행 중 오류가 발생하면 [실패] 로그를 출력하고 다음 파트로 넘어간다.


        5-2-5. read_part_excels

            1) 각 파트의 output 엑셀 파일 존재 여부를 확인한다.

            2) 파일이 없으면 [스킵] 로그를 출력한다.

            3) 파일이 있으면 pandas.read_excel로 읽는다.

            4) normalize_df로 컬럼을 통일한다.

            5) 모든 DataFrame을 concat하여 하나의 전체 DataFrame으로 합친다.


        5-2-6. save_final_excels

            1) 실행 시각 yyyymmddhhmmss를 생성한다.

            2) 원본 파일명을 만든다.

            3) 정리본 파일명을 만든다.

            4) make_clean_df로 정리본 DataFrame을 만든다.

            5) 원본과 정리본을 각각 Excel로 저장한다.

            6) format_excel로 틀 고정, 줄바꿈, 컬럼 너비를 적용한다.


6. 이메일 처리 기준

    6-1. 이메일 추출 정규식

        6-1-1. 정규식

            [A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}


        6-1-2. 의미

            1) 영문 대소문자, 숫자, 점, 언더바, 퍼센트, 플러스, 하이픈을 허용한다.

            2) @ 뒤에는 도메인을 허용한다.

            3) 마지막 확장자는 2글자 이상 영문으로 본다.


    6-2. 원본 이메일 기준

        6-2-1. 원본 보존

            1) 원본 엑셀은 수집값을 최대한 보존한다.

            2) 빈 이메일도 남긴다.

            3) 중복 이메일도 남긴다.

            4) 이메일 형식이 아닌 값도 원본에는 남긴다.


    6-3. 정리본 이메일 기준

        6-3-1. clean_email 함수

            1) 값을 문자열로 변환한다.

            2) 앞뒤 공백을 제거한다.

            3) 빈값 또는 nan이면 빈 문자열로 처리한다.

            4) EMAIL_RE 정규식으로 첫 번째 이메일 형식만 추출한다.

            5) 이메일 끝의 마침표, 쉼표, 세미콜론, 콜론, 괄호, 따옴표를 제거한다.


        6-3-2. make_clean_df 함수

            1) 원본 DataFrame을 복사한다.

            2) 이메일 컬럼에 clean_email을 적용한다.

            3) 이메일이 빈 문자열인 row를 제거한다.

            4) _email_key 컬럼을 임시로 만든다.

            5) _email_key는 이메일을 소문자로 변환한 값이다.

            6) _email_key 기준으로 drop_duplicates를 실행한다.

            7) keep="first"로 가장 위에 나온 첫 row만 유지한다.

            8) _email_key 임시 컬럼을 삭제한다.


    6-4. 중복 제거 기준

        6-4-1. 기준

            1) 이메일이 완전히 동일하면 중복으로 본다.

            2) 대소문자만 다른 이메일도 동일 이메일로 본다.

            3) 가장 먼저 수집된 row만 살아남는다.

            4) 나중에 나온 동일 이메일 row는 정리본에서 제거한다.


        6-4-2. 예시

            원본 순서:

                1행  sample@dongguk.edu
                2행  SAMPLE@dongguk.edu
                3행  sample@dongguk.edu

            정리본 결과:

                1행 sample@dongguk.edu 만 유지

    7-2. 대학 / 학과 목록 수집

        7-2-1. 기본 영역

            1) START_URL에 접속한다.

            2) BeautifulSoup으로 HTML을 파싱한다.

            3) .univ .item 목록을 반복한다.

            4) 각 item 안의 h3.depart_tit 텍스트를 대분류로 사용한다.

            5) item 안의 ul.univ_list a.link를 학과 링크로 사용한다.

            6) a.link 안의 span 텍스트를 중분류/소분류 학과명으로 사용한다.

            7) a.link의 href를 학과 URL로 사용한다.


        7-2-2. URL 없는 학과

            1) ul.univ_list div.link 구조는 URL이 없는 학과로 본다.

            3) URL과 교수URL은 빈값으로 둔다.


        7-2-3. 대분류 자체 row 제외

            1) 대분류 = 중분류 = 소분류가 되는 단과대학 자체 row는 만들지 않는다.

            2) 학과 단위 데이터만 수집한다.

            8) 게시판형 URL이면 score를 높게 준다.

            9) 가장 score가 높은 후보 URL을 공지 URL로 사용한다.

            1) 그누보드 bo_table URL이면 page=번호 방식으로 페이지를 만든다.

            2) iBuilder /bbs/data/list.do URL이면 pageIndex=번호 방식으로 페이지를 만든다.

            3) 일반 동국대 article list URL이면 아래 형식으로 만든다.

                /article/.../list?pageIndex=N&searchCondition=&searchKeyword=&category_cd=


        7-8-6. 상세 URL 수집 방식

            처리 함수:
                get_notice_detail_targets

            처리 구조:
                1) table.board tbody a 안의 /detail/ 링크를 수집한다.
                2) onclick="goDetail(번호)" 구조를 처리한다.
                3) 그누보드 bo_table + wr_id 구조를 처리한다.
                4) iBuilder javascript:fn_view('BM...', 'BD...', '') 구조를 처리한다.
                5) 중복 상세 target은 key 기준으로 제거한다.


        7-8-7. iBuilder fn_view 처리

            대상:
                글로벌무역학과 게시판

            목록 링크 예시:
                javascript:fn_view('BM0000000036','BD0000000023','');

            처리 방식:
                1) 목록 페이지에 다시 접속한다.
                2) Selenium execute_script로 fn_view(arguments[0], arguments[1], arguments[2])를 실행한다.
                3) 상세 페이지로 이동한 뒤 driver.page_source를 파싱한다.

11. 실행 방법

    11-1. 패키지 설치

        11-1-1. 설치 명령어

            pip install selenium pandas beautifulsoup4 openpyxl requests


    11-2. 실행 명령어

        11-2-1. 실행

            python dongguk_email_all_in_one.py


    11-3. 실행 전 확인

        11-3-1. Chrome / ChromeDriver

            1) Selenium을 사용하므로 Chrome 실행 환경이 필요하다.

            2) Selenium Manager가 동작하는 환경이면 별도 ChromeDriver 지정 없이 실행 가능하다.

            3) 회사 PC 또는 보안 PC에서는 ChromeDriver 권한 문제가 생길 수 있다.


        11-3-2. 네트워크

            1) 동국대 사이트 접속이 가능해야 한다.

            2) 일부 학과 사이트는 http만 가능한 경우가 있다.

            3) 광고홍보학과는 https가 아니라 http 예외 URL을 사용한다.


    11-4. 실행 후 확인

        11-4-1. 생성 파일

            1) 중간 결과 3개가 생성되는지 확인한다.

            2) 최종 원본 엑셀 1개가 생성되는지 확인한다.

            3) 최종 정리본 엑셀 1개가 생성되는지 확인한다.


        11-4-2. 정리본 확인

            1) 이메일 빈값이 없어야 한다.

            2) 이메일 형식이 아닌 값이 없어야 한다.

            3) 같은 이메일이 여러 번 나오지 않아야 한다.

            4) 같은 이메일이 있었던 경우 원본 기준 가장 위 row만 남아야 한다.

    14-1. 실행 검수

        1) 동국대_통합_임시 폴더가 생성되는지 확인한다.

        2) 임시 폴더 안에 파트 py 3개가 생성되는지 확인한다.

        3) 대학 / 대학원 / 산학협력단 실행 로그가 모두 출력되는지 확인한다.

        5) 최종 원본 엑셀과 정리본 엑셀이 생성되는지 확인한다.


    14-2. 원본 엑셀 검수

        1) 이메일이 없는 row도 남아 있는지 확인한다.

        2) 교수 목록을 못 찾은 학과의 빈 row가 남아 있는지 확인한다.

        3) 중복 이메일 row도 남아 있는지 확인한다.

    14-3. 정리본 엑셀 검수

        2) 이메일 형식이 아닌 row가 없는지 확인한다.

        3) 이메일 중복이 제거되었는지 확인한다.

        4) 동일 이메일은 원본에서 가장 먼저 나온 row가 유지되었는지 확인한다.

16. 주의사항

    16-1. 사이트 구조 변경

        1) 동국대 사이트 HTML 구조가 바뀌면 selector를 수정해야 한다.

        2) 교수 구조가 바뀌면 parse_professors 계열 함수를 수정해야 한다.

        3) 공지 구조가 바뀌면 get_notice_detail_targets 또는 board_view selector를 수정해야 한다.


    16-2. http / https

        1) 일부 학과 사이트는 https 접속 시 데이터가 안 나올 수 있다.

        2) 광고홍보학과는 http URL을 강제로 사용한다.


    16-3. 실행 시간

        1) Selenium으로 많은 페이지를 접속하므로 시간이 오래 걸릴 수 있다.

        2) 학교 사이트 응답 속도에 따라 실행 시간이 달라진다.

        3) 공지 탐색 페이지 수를 늘리면 실행 시간이 증가한다.


    16-4. 원본과 정리본 차이

        1) 원본은 검수용 전체 데이터이다.

        2) 정리본은 이메일 기준으로 실제 사용할 수 있는 데이터이다.

        3) 원본 row 수가 정리본 row 수보다 많은 것이 정상이다.


17. AI 재개발 지시 요약

    17-1. AI에게 이 설명서만 전달할 때 핵심 요구사항

        1) Python으로 작성한다.

        2) Selenium, BeautifulSoup, pandas, openpyxl, requests를 사용한다.

        3) 대학 / 대학원 / 산학협력단 3개 영역을 수집한다.

        4) 최종 컬럼은 구분, 대분류, 중분류, 소분류, 직위, 업무, 이름, 이메일, URL, 학과URL, 교수URL 순서로 맞춘다.

        5) 원본 엑셀은 빈 이메일과 중복 이메일을 제거하지 않는다.

        6) 정리본 엑셀은 이메일 빈값 제거, 이메일 형식 아닌 값 제거, 이메일 소문자 기준 중복 제거를 한다.

        7) 동일 이메일은 가장 먼저 나온 row만 남긴다.

        8) 교수는 직위=교수, 업무=교육·연구로 저장한다.

        9) 행정 이메일은 소분류=행정, 업무=행정, 이름=빈값으로 저장한다.

        10) 대학 공지 행정 이메일은 교수 이메일과 겹치면 제외한다.

        11) 공지 상세에서 @dongguk 포함 이메일을 찾으면 해당 학과 공지 탐색을 중지한다.

        12) 광고홍보학과는 http URL 예외를 적용한다.

        13) 글로벌무역학과 iBuilder fn_view 게시판 상세 이동을 처리한다.

        14) 산학협력단은 requests로 정적 테이블을 수집한다.

        15) 엑셀은 첫 행 고정, 줄바꿈, 컬럼 너비 자동 조정을 적용한다.
